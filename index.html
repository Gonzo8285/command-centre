Bridge v2 — what it does, in plain English

design summary

The problem it solves

  • v1 bridge is async file drops via Drive — fine for messages, useless for live handoff.
  • PPC's sandbox can't reach the LAN, only the public internet — so the link must be HTTPS-routable.
  • You wanted "accessible from anywhere" and shared compute between us.

How it works

  • Transport: Tailscale + Funnel gives each host a https://<host>.<tailnet>.ts.net URL.
  • Daemon: Node, ~370 LOC, zero npm deps. Runs as Scheduled Task at logon — no admin needed.
  • Auth: Bearer token in shared/.bridge-token. Rotate by overwriting the file.
  • Fallback: If heartbeat > 5 min, callers automatically drop back to v1 file bridge.

Endpoints

  • /health — unauth, returns 200 when alive.
  • /resources — auth, host capacity snapshot.
  • /tools — auth, what this host can do right now.
  • /msg — auth, accepts v1 + v2 message shapes.
  • /delegate — auth, queues a job; bounces if overloaded.
  • /inbox — auth, drains queue.

Phase plan

  • A — Bridge online on both hosts (waiting on you).
  • B — PPC drains 151's queue: scheduled tasks first, then FastForward builds, then the rest.
  • C — Resume rest of portfolio per the phase-gate.
